## Context

当前新闻聚合系统有 16 个爬虫失败，分布在三个主要类别：GitHub（3个）、编程语言官方博客（8个）、技术媒体（6个）。这些爬虫使用不同的技术栈（requests、Selenium、BeautifulSoup），失败原因可能包括：网站结构变化、反爬虫机制、网络超时、API 限流等。

当前系统缺乏统一的错误处理和健康监控机制，导致爬虫失败时难以快速定位问题。

## Goals / Non-Goals

**Goals:**
- 诊断并修复所有 16 个失败的爬虫
- 建立统一的错误处理和重试机制
- 添加爬虫健康检查，及时发现失败
- 提升系统的可观测性和可维护性

**Non-Goals:**
- 不重构整个爬虫架构
- 不添加实时告警系统（仅记录和报告）
- 不优化爬虫性能（专注于修复失败）

## Decisions

### 1. 分阶段诊断和修复策略
**决策**：按类别逐个诊断和修复，而非一次性修复所有爬虫
**理由**：不同类别的爬虫失败原因可能不同，分阶段可以更好地识别共性问题
**替代方案**：并行修复所有爬虫 - 但可能导致重复工作和遗漏共性问题

### 2. 统一的错误恢复基类
**决策**：创建 BaseCrawlerWithRetry 基类，提供统一的重试和错误处理
**理由**：避免在每个爬虫中重复实现相同的错误处理逻辑
**替代方案**：使用装饰器模式 - 但基类继承更符合现有代码结构

### 3. 健康检查机制
**决策**：在每次爬取后记录成功/失败状态到 MongoDB，生成健康报告
**理由**：利用现有 MongoDB 基础设施，无需引入新的监控系统
**替代方案**：使用独立的监控服务 - 但增加了系统复杂度

### 4. 失败原因分类
**决策**：将失败原因分为：网络错误、解析错误、API 限流、反爬虫、其他
**理由**：便于统计分析和针对性修复

## Risks / Trade-offs

- **[风险] 网站持续变化** → 建立定期检查机制，及时发现新的失败
- **[风险] 反爬虫机制升级** → 添加 User-Agent 轮换、请求延迟等对策
- **[权衡] 重试次数与执行时间** → 默认重试 3 次，可配置
- **[权衡] 健康检查数据量** → 仅保留最近 30 天的健康记录

## Migration Plan

1. 第一阶段：添加基础设施（错误恢复基类、健康检查）
2. 第二阶段：按类别修复爬虫（GitHub → Language → Tech News）
3. 第三阶段：验证修复效果，生成健康报告
4. 回滚策略：保留原有爬虫代码，通过配置开关控制新旧版本
